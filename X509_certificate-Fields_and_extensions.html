X509 certificate - Fields & extensions

Date dim. 03 avril 2016 Modified mar. 19 avril 2016 Tags Certificate

For X509 certificate, the fields and extension are defined by the RFC5280. As well, the CA/B Forum (Certificate Authority & Browser Forum) defines best pratice. This post is a memo to remind the main fields and extension.

Basic certificate field

Field Description
Subject The subject must be populated with a non-empty distinguished name. The DN must be defined, at least: Common Name, Organization Name and Organization Unit
Serial Number This is certificate identifier. So it must be unique for each certificate issued by a given CA. According to best practice, the first serial number must be defined randomly. Then, the number must be increased randomly.

Certificate extension

Extension Description
Subject Key Identifier This identifier is a way to identify the public key. This identifier should be derived from the public key (for example: with SHA-1)
Authority Key Identifier This extension must be equal to the "Subject Key Identifier" of the Certification Authority
Key Usage This extension defines the purpose of the key contained in the certificate
Extended Key Usage This extension defines the purpose of the key in addition to or in place of the basic purposes indicated in the key usage extension
Basic Constraints This extension identifies whether the subject of the certificate is a CA and the maximum depth of valid certification paths that include this certificate
Authority Information Access This extension indicates how to access information and services of the CA (on-line validation services and CA policy data). Example: "OCSP;URI:http://ocsp.example.com/RootCA/"
crlDistributionPoints This extension defines how to obtain CRL. Example: "URI:http://www.example.com/RootCA.crl"

Critical or Non critical: A software must reject the certificate if it encounters a critical extension that it doesn't recognize, or a critical extension that contains information that it can't process. A non-critical extension may be ignored if it is not recognized, but must be processed if it is recognized.
